# 《幻影微屋》产品商业需求文档（BRD）

**—— 基于AI与UGC的微缩3D虚拟场景引擎**

## 1. 产品概述

- **产品定位**：一款专注于生成与编辑"10cm微缩比例3D世界"的数字引擎软件。
- **最终愿景**：作为"灵魂"注入未来的桌面物理显示设备，将只存在于数字世界的完整场景（非单一人物），转化为真实桌面的物理陪伴物。
- **一句话描述**：不只是造一个会动的虚拟人，而是造一个有风景、有家具、有氛围的微缩平行宇宙。

## 2. 市场痛点与竞品分析

- **对标竞品 Project AVA 的核心缺陷**：AVA 仅解决了"单一虚拟人的物理底座驱动"问题，但人物处于**"无背景、无场景"的裸奔状态**，缺乏环境叙事能力，本质上仍是"会动的手办"，而非"一个世界"。
- **痛点 1：屏幕隔离感未根除**：即便有AVA，仍需通过手机屏幕去观察交互，无法实现物理层面的"被动陪伴"。
- **痛点 2：3D场景创作门槛极高**：现有AI生3D多针对单一物体（人物/道具），缺乏对"包含透视、光影、多物体组合的完整场景"的生成能力。
- **痛点 3：缺乏微缩生态的UGC土壤**：用户无法以极低的成本，像玩《模拟人生》或《动森》那样，随心所欲地布置自己的虚拟微缩房间。

## 3. 核心产品策略（软件先行）

- **策略一："人+景"一体化强制绑定**。软件生成的任何数字人，必须处于一个特定的3D环境中。禁止输出"透明底/纯色底"的单一人物模型。
- **策略二：尺寸与风格的强制物理预判**。虽然现阶段是纯软件，但所有生成的3D资产（场景、人物、家具）在系统内的比例尺必须严格锁定在 **10cm潮玩盲盒比例**，且强制锁定 **Q版/低多边形/微缩黏土风**，为未来极低算力的硬件渲染铺路，同时利用微缩滤镜掩盖AI精度瑕疵。
- **策略三：硬件解耦，定义标准接口**。现阶段不纠结硬件光学方案，软件端只负责打造极致的微缩数字内容，并对外提供标准的 `.glb` 场景导出与局域网推送接口，等待未来合适的物理显示载体。

## 4. 核心功能规划（软件端）

### 4.1 AI 场景生成引擎（核心卖点解决）

- **文/图生全景**：用户输入"赛博朋克的小巷"或上传参考图，AI直接生成带有纵深感的微缩3D场景（包含地面、墙壁、远景、环境光）。
- **环境氛围生成**：不仅仅是硬体建筑，包含场景内的动态元素（如微缩的飘落樱花、缓缓流动的水面、闪烁的霓虹灯牌）。

### 4.2 微缩 UGC 编辑器（核心护城河）

- **上帝视角编辑**：提供俯视/45度角视角的2.5D编辑界面。
- **模组化拖拽**：提供海量微缩家具库（桌子、床、路灯、杯子），用户可自由拖拽进AI生成的场景中进行排版组合。
- **物理属性微调**：支持调整家具的摆放角度、开关灯状态、颜色等。

### 4.3 数字人入驻与多智能体互动

- **放入场景**：将AI生成的Q版小人放置在编辑好的场景中，可自由选择站位（坐在沙发上、站在窗边）。
- **多角色同居**：一个场景内支持放置多个数字人。
- **环境交互触发**：设定简单的条件逻辑（如：时间到晚上8点，场景灯光变暗，小人A走到小人B的床边）。

### 4.4 "投射准备"模块（为未来硬件预留）

- **画框预览模式**：在电脑屏幕上模拟一个物理相框的边界，让用户以"未来在桌面上看它"的视角来预览场景效果。
- **一键打包���送**：将复杂的场景（包含模型、贴图、骨骼、灯光烘焙）打包为一个轻量的 `.glb` 文件，通过局域网发送给指定的接收端（目前的测试端可以是另一个手机/电脑屏幕，未来是硬件盒子）。

## 5. 商业模式（基于数字资产）

- **AI算力消耗**：按次收费生成高级场景或高级精度人物。
- **虚拟建材商城（主要收入）**：售卖非基础的微缩家具包、特殊氛围灯光包、特殊地形包（如"废弃教室扩建包"、"魔法森林地板包"）。
- **创作者集市抽成**：UGC创作者上传自己用AI+手工调整出的精美微缩场景/家具，其他用户付费下载，平台抽成。

## 6. 技术关键点与成本控制（纯软件端）

- **模型轻量化（必做）**：必须在软件后端强制部署自动减面算法，将AI生成的复杂场景（可能几十万面）压缩到 **5万面以内**，确保未来在廉价硬件上能流畅跑动。
- **光影烘焙前置**：因为未来硬件极大概率无法实时演算光影，软件端在用户点击"保存/推送"时，必须将场景的光影效果"烘焙"成贴图，直接画在模型上，牺牲动态光影，换取极低的渲染性能消耗。
- **API成本控制**：3D场景生成成本较高（预估单次1-3元），需设计"基础白模场景免费 + 高级氛围/细节收费"的策略。

## 7. MVP（最小可行性产品）执行路径

**目标：不写一行硬件代码，纯靠软件证明"微缩场景DIY"的吸引力。**

1. **开发环境**：基于 WebGL（如 Three.js）或 Unity 开发一个网页端/PC端编辑器。
2. **砍掉AI生成（前期）**：MVP阶段为了节省API成本和避开AI场景生成的不稳定性，**内置 3-5 套手工制作的高质量基础微缩场景**（如：一个日式和室、一个赛博吧台）。
3. **聚焦UGC核心**：提供 50 个基础微缩家具模型，让测试用户在这个网页里像玩《模拟人生》一样拖拽布置房间，并把小人放进去。
4. **验证指标**：观察用户在没有硬件、只有屏幕的情况下，是否愿意花 15 分钟去布置一个微缩房间。如果用户觉得布置的过程本身有乐趣，且布置完觉得"很有成就感想截图分享"，则证明软件的"场景DIY护城河"逻辑成立，再引入AI生成和硬件对接。
